Use Case Scenarios Report

1. Use Case Scenarios

Example Use Case Scenario: Customer Creates an Account

Use Case Name: Create Account

Preconditions:

  • The customer is not yet registered in the system.
  • The customer is on the registration page.

Postconditions:

  • The customer is registered in the system.
  • The customer can log in to the system with the new credentials.

Actors: Customer

Main Flow:

  1. The Customer navigates to the "Create Account" option on the home page.
  2. The System shows the account creation form.
  3. The Customer enters the required information (e.g., name, email, password) in the form.
  4. The System validates the input fields.
  5. The Customer submits the registration form.
  6. The System checks if the email address is already registered.
  7. The System creates a new account for the Customer.
  8. The System sends a confirmation email to the Customer.
  9. The System displays a success message and redirects the Customer to the login page.

Alternate Flows:

  • Alternate Flow 1 (Invalid Email Format):
    • Step 3: The Customer enters an invalid email address (e.g., missing "@" or ".com").
    • System Reaction: The System displays an error message asking the Customer to correct the email format. The process returns to Step 3.
  • Alternate Flow 2 (Email Already Registered):
    • Step 6: The System detects that the email is already registered.
    • System Reaction: The System displays an error message and prompts the Customer to use a different email address. The process returns to Step 3.
  • Alternate Flow 3 (Weak Password):
    • Step 3: The Customer enters a weak password (e.g., less than 8 characters or lacking special characters).
    • System Reaction: The System displays a message requesting the Customer to create a stronger password. The process returns to Step 3.

Example Use Case Scenario: Customer Makes a Payment

Use Case Name: Customer Makes a Payment

Preconditions:

  • The Customer has selected products and added them to the shopping cart.
  • The Customer has logged in to the system.

Postconditions:

  • The System processes the payment.
  • The Customer receives a payment confirmation.
  • The Order is marked as "Paid" in the system.
  • The System updates the inventory.

Actors: Customer, System

Main Flow:

  1. The Customer navigates to the "Cart" page.
  2. The System displays a summary of the items in the cart.
  3. The Customer clicks on the "Proceed to Checkout" button.
  4. The System requests payment details (credit card, PayPal, etc.).
  5. The Customer enters the payment details and submits the form.
  6. The System processes the payment.
  7. The System confirms that the payment is successful.
  8. The System displays a payment confirmation page.
  9. The System updates the inventory and prepares the order for dispatch.

Alternate Flows:

  • Alternate Flow 1 (Payment Declined):
    • Step 6: The System detects that the payment is declined (e.g., insufficient funds).
    • System Reaction: The System displays a payment error message and prompts the Customer to try again with a different method. The process returns to Step 4.
  • Alternate Flow 2 (Invalid Payment Details):
    • Step 5: The Customer enters invalid payment details (e.g., expired credit card).
    • System Reaction: The System displays an error message and requests the Customer to update the payment details. The process returns to Step 5.

Example Use Case Scenario: Seller Adds a Product

Use Case Name: Add/Remove Product

Preconditions:

  • The Seller is logged into the system.
  • The Seller has access to the "Seller Dashboard."

Postconditions:

  • A new product is added to the system.
  • The product is available for customers to browse and purchase.

Actors: Seller

Main Flow:

  1. The Seller navigates to the "Add Product" section on the dashboard.
  2. The System displays a form for entering product details (name, price, description, etc.).
  3. The Seller enters the required product details.
  4. The Seller uploads images of the product.
  5. The System validates the product details.
  6. The Seller submits the form to add the product.
  7. The System adds the product to the catalog.
  8. The System confirms that the product has been added.

Alternate Flows:

  • Alternate Flow 1 (Missing Required Fields):
    • Step 3: The Seller misses one or more required fields (e.g., price or description).
    • System Reaction: The System prompts the Seller to complete all required fields before submission. The process returns to Step 3.
  • Alternate Flow 2 (Invalid Image Format):
    • Step 4: The Seller uploads an unsupported image format.
    • System Reaction: The System displays an error message and requests the Seller to upload a valid image. The process returns to Step 4.

2. Structure of the Use Case Scenarios

Use Case Name: Name of the use case (e.g., Create Account, Payment Process).

Preconditions: Conditions that must be satisfied for the use case to start.

Postconditions: Conditions that will be satisfied after the use case is executed successfully.

Actors: The actors involved in the use case (e.g., Customer, System, Seller).

Main Flow: The main sequence of steps describing the execution of the use case.

Alternate Flows: Possible alternative flows that handle deviations from the main flow (e.g., invalid input, system errors).

Language: English
Preconditions:
  • The user must have an active account on shopping platform
  • The user must have placed an order (delivered or in progress)
  • The user has logged into the system
Postconditions:
  • Petition is submitted and acknowledged by the system
  • The issue is logged in the system and queued for resolution
  • User receives a notification that the petition is successfully sent.
Actors:
  • Customer
  • Customer Support
Main flow:
  1. The User (customer) logs into the platform
  2. Customer navigates to the Orders section
  3. Customer selects the order for which they want to file a report (e.g., damaged item, wrong product)
  4. Customer clicks the Report Issue button.
  5. The system displays a form for the user to fill in details about the issue (e.g., issue type, description)
  6. Customer submits the report petition
  7. The system generates a petition ID
  8. System sends a confirmation mail or notification to the user with details of the petition
  9. The issue is forwarded to the backend team for resolution
Alternate flow:
  • 1. User encounters an error during submission (due to missing information or a technical issue). The customer can re-submit the form after addressing the issue. The process returns to Step 5. END FLOW → 5
  • 2. User cancels the report petition (user can do it before submitting the petition). System discards the previous input and redirects the user back to the Orders section; the process ends before returning to the Main Flow. END FLOW
  • 3. No relevant order found (e.g. the order was canceled). The system displays an error message instead of a notification of confirmation for successful submission. The process returns to Step 3. END FLOW → 3

  8. The system updates the wishlist based on the changes made by the customer

5. Conclusion

The use case scenarios outlined in this report provide a comprehensive understanding of the various interactions between users and the system in the context of online shopping. From account creation to issue reporting, each scenario illustrates the critical steps and potential error handling that ensure a smooth user experience. By considering both the main and alternate flows, these scenarios allow developers and stakeholders to identify and address possible challenges, thereby enhancing system functionality and customer satisfaction. Stakeholders are encouraged to utilize these findings to refine product features and optimize user engagement methods.
