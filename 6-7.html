Class Diagrams Report

1. Class Diagrams

To create a Class Diagram based on the system described in your PlantUML code and the task details, we will define several classes based on the use cases and their interactions. Each actor (Customer, Seller, Admin, and Support Agent) will correspond to certain classes, and these classes will have attributes and methods associated with them.

2. Identified Classes

  • Customer: Represents the customer in the system.
  • Seller: Represents the seller.
  • Admin: Represents the administrator.
  • SupportAgent: Represents the support agent.
  • Product: Represents the product that the customer can browse and purchase.
  • Order: Represents the order made by the customer.
  • Payment: Represents the payment process.
  • Inventory: Represents the inventory managed by the seller.
  • SupportTicket: Represents a support ticket created by the customer.

3. Class Diagram Components

  • Attributes: The data that each class holds (e.g., CustomerID, ProductName, Quantity).
  • Methods: Functions that can be called on an object of the class (e.g., createAccount(), addToCart()).
  • Relations:
    • Association: Represents a direct relationship between two classes (e.g., Customer uses Product).
    • Aggregation: Represents a "has-a" relationship, where a class contains objects of another class (e.g., Order aggregates Product).
    • Composition: Represents a stronger form of aggregation, indicating that one class depends on the existence of another (e.g., Order is composed of Product).
    • Generalization: Inheritance relationships (e.g., a User class could be a generalization of Customer and Seller).

4. Explanation

  • User Class: An abstract base class for common attributes like userID, name, and email, with common methods like createAccount() and login().
  • Customer, Seller, Admin, SupportAgent: These classes inherit from the User class, representing specific types of users.
  • Product: Contains attributes like productID, productName, and price, and a method to display product details.
  • Order: Contains information about an order made by the customer, such as orderID, customerID, and orderDate. It aggregates Product objects, meaning an order contains multiple products.
  • Payment: Represents the payment process for an order, with methods like processPayment() and refundPayment().
  • Inventory: The seller's inventory, which contains a list of products and their stock quantity.
  • SupportTicket: Represents a support ticket raised by a customer for a complaint or issue, with methods to open and close the ticket.

5. Relations

  • Generalization (<|--): Customer, Seller, Admin, and SupportAgent all inherit from the User class.
  • Association (-->): The Order class contains Product objects (one order can have multiple products).
  • Aggregation: Order aggregates Product, meaning the Order class depends on Product but the Product can exist without the Order.
  • Composition: The Payment class processes payments for specific Order instances. If the order is deleted, the payment related to that order may not exist anymore.
